# PLAYBOOKMD — P3-R1: PATIENT PROFILE + OPPORTUNITY PANEL

## PROMPT REFERENCE
Code: P3-R1 | Phase 3, Revision 1
Previous phases: P1 (foundation + shell), P2 (pipeline kanban)

---

## OVERVIEW

Phase 3 builds two interconnected things:
1. **Opportunity Detail Panel** — a slide-out panel that opens when clicking a pipeline card. Lightweight, focused, action-oriented.
2. **Patient Profile Page** — a full-page deep dive into a patient's complete history. Accessed from the panel or from the Patients list.

Do NOT build these as separate unrelated screens. They are two levels of the same information hierarchy: Panel (quick context + actions) → Full Profile (complete record).

Run on HIGH AUTONOMY. Follow established design system throughout. Make bold visual choices on data presentation — especially quotes/invoices. Avoid generic table layouts where something more visual would serve better.

---

## PART 1: OPPORTUNITY PANEL (SLIDE-OUT)

### Trigger
Clicking any pipeline card opens this panel. The pipeline board remains visible and interactive behind the panel.

### Panel Behavior
- Slides in from the right edge of the viewport
- Width: 400px on desktop, full width on mobile
- Does not cover the entire board — board remains partially visible to the left
- Clicking a different pipeline card while panel is open: panel stays open, content swaps with new opportunity data (no close/reopen animation)
- Close: X button top right, or press Escape
- Background: white
- Left border: `3px solid [stage accent color]` — visual connection to the column this card came from
- Box shadow: `0 0 40px rgba(0,0,0,0.15)` — lifts panel above board
- Z-index: 50

### Panel Header

**Patient name:** 18px, weight 700, `--text-primary`

**Tag pills** (immediately below name, in a flex-wrap row):
Tags are stored as free-text strings on the patient record. Render each as a small pill.
Special rendering for known tag values:
- "VIP" → emerald background `#D1FAE5`, dark emerald text `#065F46`, crown emoji prefix 👑
- "Discharged" → red background `#FEE2E2`, dark red text `#991B1B`, ⛔ emoji prefix — make this visually prominent, it's a warning
- Any other tag → gray `var(--bg-tertiary)` background, `--text-secondary` text

Tags are editable: clicking "+ Add Tag" opens a small inline input to type a new tag, press Enter to add. Clicking an existing tag shows an X to remove it. Store tags as a string array on the Patient model — add `tags String[] @default([])` to Patient model if not present, run migration.

**Two-column assignment row** (below tags):
```
👤 Coordinator: Jessica M.        🩺 Provider: Dr. Martinez
```
- Label: 10px uppercase `--text-muted`
- Value: 13px weight 600 `--text-primary`
- Both are dropdown-editable inline — clicking the value opens a select of practice users (coordinator) or providers (provider)
- Coordinator = who sold it (sales performance tracking)
- Provider = who performs the procedure — could be surgeon, nurse, esthetician, dermatologist — do not assume doctor

**Opportunity details row:**
- Procedure: 14px, weight 600
- Stage badge: small pill with stage accent color
- Estimated value: Geist Mono, 14px, weight 700, emerald
- Lead source: 12px, `--text-muted`, italic

### Appointment Storyline

Section label: "APPOINTMENTS"

Compact chronological list of all appointments linked to this opportunity (past and future).

**Past appointments** (before today):
- Grayed out: `--text-muted` for all text
- Status badge (small, right-aligned):
  - Completed: emerald dot + "Done"
  - No Show: red dot + "No Show"
  - Cancelled: gray dot + "Cancelled"
- Date in Geist Mono

**Future appointments** (today or later):
- Full color, not grayed
- Date in Geist Mono, emerald if within 7 days

**Visual design:** Keep this compact. Each appointment is a single line — date + type + provider + status badge. If more than 3 appointments, show first 2 and "View all X" link.

**"Book Appointment" button:** Always visible below the list, even if upcoming appointments exist.
- Emerald outlined button, `CalendarPlus` icon + "Book Appointment"
- On click: placeholder toast "Appointment booking coming in Phase 4 — patient and opportunity will be pre-linked"

### Forms Status

Section label: "FORMS"

Show 2-3 key forms per opportunity. Seed with: Intake Form, Surgical Consent (if applicable), Financial Agreement.

Each form as a single compact row:
```
[icon] Intake Form           ● Not Sent    [Send]
[icon] Surgical Consent      ● Sent        [Remind]
[icon] Financial Agreement   ✓ Completed   [View]
```

**Status:**
- Not Sent: gray dot, gray text, "Send" action button (placeholder)
- Sent: amber dot, amber text, "Remind" action button (placeholder)
- Completed: emerald checkmark, emerald text, "View" action button (placeholder)

Store on `OpportunityForm` model:
```prisma
model OpportunityForm {
  id            String    @id @default(cuid())
  opportunityId String
  formName      String
  status        String    @default("NOT_SENT") // NOT_SENT | SENT | COMPLETED
  sentAt        DateTime?
  completedAt   DateTime?
  createdAt     DateTime  @default(now())
  opportunity   Opportunity @relation(fields: [opportunityId], references: [id])
}
```
Run migration. Seed varied statuses across opportunities.

## API ROUTES

`GET /api/patients` — paginated list with computed fields
`GET /api/patients/[id]` — full patient with sidebar data
`GET /api/patients/[id]/timeline?type=&page=&limit=20` — merged sorted feed
`POST /api/patients/[id]/notes` — creates Activity NOTE
`GET /api/patients/[id]/financial` — quotes, invoices, totals
`GET /api/opportunities/[id]/panel` — opportunity with patient, appointments, quotes, invoices, forms
`POST /api/activities` — creates any activity record
`PATCH /api/patients/[id]/tags` — updates tags string array
