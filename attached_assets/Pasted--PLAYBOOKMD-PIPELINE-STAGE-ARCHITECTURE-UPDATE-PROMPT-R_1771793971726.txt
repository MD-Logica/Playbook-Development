This is a data migration + UI update. The board should reflect the new stages after this runs.

---

## DESIGN PRINCIPLES FOR STAGE CONFIGURATION

Before specifying the stages, the following principles must be respected:

1. **Unresponsive and Responsive are separate stages** — not combined into "In Follow-Up." This is intentional. Reporting on where leads are lost (from Unresponsive vs. from Responsive) reveals lead quality. A lead lost from Unresponsive = bad quality lead (couldn't reach them). A lead lost from Responsive = good quality lead (something went wrong in the conversation). These require different corrective actions and must be distinguishable in reports.

2. **Rotting badges measure staff inactivity, not patient inactivity.** A coordinator can call an unresponsive patient daily — the card won't rot — but the patient is still ghosting. The Unresponsive stage handles patient inactivity separately from the rotting indicator.

3. **No duplicate stages.** "Quote Sent" and "Decision Pending" are the same moment: quote is in the patient's hands, ball is in their court. Consolidate into one.

4. **Expiring Quotes is an automation-triggered stage.** It is not manually moved to. It requires automation logic (see below).

5. **Keep total stage count reasonable.** More than 8-9 stages on a board becomes cognitively overwhelming. Each stage must earn its existence with a distinct workflow action.

## EXPIRING QUOTE STAGE — AUTOMATION LOGIC

This stage is populated automatically, not by manual card moves. Build the following automation:

### Trigger condition (check nightly via cron or on-demand):
```
Quote exists for an opportunity
AND Quote status = PENDING (not yet accepted or expired)
AND Quote has an expiresAt date
AND Today >= (expiresAt - 20 days)  ← within 20 days of expiration
AND No Invoice linked to this opportunity (patient hasn't committed)
AND Opportunity is not already in "Expiring Quote", "Deposit Paid", or later stage
```

### Action:
- Move the opportunity to the "Expiring Quote" stage
- Create an Activity record: `{ type: "STAGE_CHANGE", content: "Auto-moved to Expiring Quote — quote expires on [date]", ... }`
- The coordinator sees this in the next morning's view and knows to call

### Cron schedule:
Run this check daily at 8:00 AM practice local time (or UTC if timezone not yet configured).

### API endpoint:
`POST /api/cron/check-expiring-quotes`
- Protected — only callable by cron or internal admin
- Returns: `{ moved: number, skipped: number }`

### Seed data for testing:
Create at least 2 opportunities with:
- Status in "Quote Sent" stage
- A quote with expiresAt within the next 20 days
- No invoice linked
So that running the cron immediately moves them to "Expiring Quote" and the feature is visibly testable.
