## CRITICAL DATA MODEL CHANGES FIRST

Before building any UI, make these schema changes to `prisma/schema.prisma`:

### 1. Link Appointments to Opportunities
Add an optional relationship between Appointment and Opportunity:

```prisma
model Appointment {
  // ... existing fields ...
  opportunityId String?
  opportunity   Opportunity? @relation(fields: [opportunityId], references: [id])
}

model Opportunity {
  // ... existing fields ...
  appointments  Appointment[]
}
```

### 2. Add rotting threshold to PipelineStage
```prisma
model PipelineStage {
  // ... existing fields ...
  rottingThresholdHours Int @default(168) // 7 days default
  color                 String? // hex color for stage column header accent
  automationTrigger     String? // event that auto-moves cards into this stage
}
```

### 3. Add Pipeline Automation Rules table
```prisma
model PipelineAutomationRule {
  id            String   @id @default(cuid())
  practiceId    String
  pipelineId    String
  fromStageId   String?  // null means "any stage"
  toStageId     String
  triggerEvent  String   // "APPOINTMENT_CREATED" | "CONSULT_COMPLETED" | "QUOTE_SENT" | "QUOTE_ACCEPTED" | "DEPOSIT_PAID" | "SURGERY_COMPLETED"
  isActive      Boolean  @default(true)
  createdAt     DateTime @default(now())
  
  practice      Practice  @relation(fields: [practiceId], references: [id])
  pipeline      Pipeline  @relation(fields: [pipelineId], references: [id])
}
```

### 4. Update Opportunity with pipeline position tracking
```prisma
model Opportunity {
  // ... existing fields ...
  lastStageMovedAt DateTime?  // when was this card last moved to a new stage
  isArchived       Boolean    @default(false)
}
```

After schema changes, run:
```bash
npx prisma migrate dev --name phase2-pipeline-relationships
npx prisma generate
```

Then update the seed data to link some existing appointments to opportunities where they logically connect (same patient, same procedure type).

- Card gets a very subtle red tint on background: `rgba(239, 68, 68, 0.03)`
- Card border becomes: `1px solid rgba(239, 68, 68, 0.3)`

## SECTION 4: DRAG AND DROP

Use `@dnd-kit/core` and `@dnd-kit/sortable` for drag-and-drop. Install if not present.

```bash
npm install @dnd-kit/core @dnd-kit/sortable @dnd-kit/utilities
```

### Implementation
- Wrap the entire board in `<DndContext>`
- Each column is a `<DroppableColumn>`
- Each card is a `<SortableCard>`
- On drag end:
  1. Optimistically update UI (move card immediately)
  2. Call `PATCH /api/opportunities/[id]/stage` with `{ stageId, position }`
  3. If API fails, revert the optimistic update and show a toast error

### Visual Feedback During Drag
- The column being dragged over gets a subtle emerald border highlight: `border: 2px solid rgba(16,185,129,0.4)`
- A drop placeholder appears in the target position — a dashed rectangle the same height as the dragged card
- The dragged card rotates slightly (2deg) and increases shadow

### Stage Transition API
**`PATCH /api/opportunities/[id]/stage`**
```typescript
// Body
{ stageId: string, position: number }

// Actions:
// 1. Update opportunity.stageId
// 2. Update opportunity.position  
// 3. Update opportunity.lastStageMovedAt = now()
// 4. Create an Activity record: type "STAGE_CHANGE", note "Moved to [stage name]"
// 5. Check if any PipelineAutomationRule triggers apply (stub for now — log but don't execute)
// 6. Return updated opportunity
```

## API ROUTES SUMMARY

All routes scope to practiceId. Use the practice resolver pattern established in Phase 1.

**`GET /api/pipeline/board?pipelineId=[id]`**
Returns pipeline with stages, each stage with its opportunities, each opportunity with patient name, assigned user, last activity date, rotting status calculated server-side.

```typescript
// Response shape
{
  pipeline: { id, name, totalValue, velocity, winRate, atRisk },
  stages: [
    {
      id, name, color, rottingThresholdHours, position,
      opportunities: [
        {
          id, patientName, procedureName, estimatedValue,
          assignedToName, leadSource, lastActivityAt,
          daysInStage, isRotting, rottingDays,
          status, nextAppointment
        }
      ],
      totalValue, count
    }
  ]
}
```

**`PATCH /api/opportunities/[id]/stage`** — move card between stages (documented above)

**`POST /api/opportunities`** — create new opportunity with patient

**`GET /api/pipeline/bottom-bar?pipelineId=[id]`** — returns today's actions count, closing this week value, stalled count
